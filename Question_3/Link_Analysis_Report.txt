Explanation: By using the -i flag with ls -l, we can observe the specific inode numbers assigned to each file. The output confirms that original.txt and hard_link.txt share the exact same inode number (419949). Conversely, sym_link.txt is assigned a completely different inode number (419950) because it exists as an independent file on the disk.

2. Comparison between hard link and symbolic link
Command Executed: file original.txt hard_link.txt sym_link.txt
Output:
original.txt:  ASCII text
hard_link.txt: ASCII text
sym_link.txt:  symbolic link to original.txt
Explanation: The file command analyzes the file type. It verifies that a hard link is functionally identical to the target file (both are ASCII text) because it acts as a direct reference to the original inode and shares its data blocks. A symbolic link, however, is identified as a special "symbolic link" file type that merely contains a text path pointing to the target location.

4. Effect of deleting the original file
Command Executed: rm original.txt followed by cat hard_link.txt and cat sym_link.txt
Output:
Important Project Data
cat: sym_link.txt: No such file or directory (os error 2)
Explanation: Removing the original file deletes its specific directory entry, but the cat command proves the hard link still works perfectly because it directly references the underlying, un-deleted data blocks. The symbolic link, however, immediately becomes a broken symlink because the exact file path it points to no longer exists.

5. Conclusions drawn from the experiment
Based on this experiment, hard links are a highly robust mechanism for preventing accidental data loss, as the actual file data remains intact on the disk until all associated hard links are deleted and the inode is fully freed. Symbolic links are highly versatile because they can point across different file systems and directories, but they are fragile because they rely entirely on the target file's path remaining unchanged.
